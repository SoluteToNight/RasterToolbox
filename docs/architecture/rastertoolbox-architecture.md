# **RasterToolbox: 开源轻量级 GIS 栅格数据管理工具架构与实施文档**

## **1\. 项目概述**

RasterToolbox 定位为“GIS 界的 Handbrake”，是一款专注于栅格数据流动、转换与轻量化的桌面级数据管理工具。本项目摒弃复杂的空间分析功能，旨在将 GDAL 底层晦涩的命令行参数转化为开箱即用的**可视化预设 (Presets)** 与**稳定的批处理队列 (Task Queue)**，解决 GIS 行业中格式转换、图像压缩、金字塔构建及切片打包的痛点。

## **2\. 技术栈定义**

本项目采用“原生 C++ 核心 \+ 现代化 GUI”的分离式架构，确保极高的 I/O 读写性能与丝滑的界面交互。

* **核心开发语言**：C++20。  
* **图形用户界面 (GUI)**：Qt 6.x (基于 **Qt Widgets** 模块，构建严谨的工业级界面)。  
* **地理空间核心引擎**：GDAL C++ API (直接调用原生 API 以实现像素级进度回调与极致性能)。  
* **构建与环境工程**：  
  * **项目构建**：CMake (3.20+)  
  * **环境与依赖管理**：Pixi (提供跨平台、项目级的 C++ 与 GDAL 依赖隔离，实现一键编译)。  
* **核心补充组件**：  
  * **JSON 解析**：nlohmann/json (用于预设配置的序列化/反序列化)。  
  * **并发控制**：QtConcurrent 结合 \<future\> (用于维护后台工作线程池)。  
  * **日志系统**：spdlog (接管并格式化 GDAL 底层报错日志)。  
  * **本地数据库**：SQLite3 (支撑 GeoPackage 规范写入及潜在的队列状态持久化)。

## **3\. 核心模块分层设计**

系统遵循严格的 MVC 架构，确保“界面展示”、“并发调度”与“底层计算”完全解耦。

### **3.1 界面表现层 (UI Presentation Layer)**

严格基于 Qt Widgets 构建，不包含耗时业务逻辑。

* **源数据控制面板**：处理文件拖拽，轻量级调用底层读取并展示栅格元数据（分辨率、波段、EPSG）。  
* **转换参数配置区**：将用户的表单选择实时映射到内存中的任务配置对象。  
* **批处理队列界面**：管理任务状态机，展示单任务与全局的多级进度条。  
* **实时日志面板**：高亮展示底层抛出的错误与警告，辅助问题排查。

### **3.2 核心调度层 (Dispatcher & Concurrency Module)**

管理多线程任务流转，防止后台高强度读写阻塞 UI。

* **任务队列管理器**：维护状态机（Pending, Running, Paused, Canceled），控制并发策略。  
* **工作线程池**：在独立线程中实例化 GDAL 封装层并执行计算，处理跨线程的安全退出信号。  
* **信号管道**：将后台 Worker 线程中的 GDAL 处理进度和 spdlog 日志转化为 Qt Signal 发送至 UI。

### **3.3 GDAL 原生封装层 (GDAL Engine Wrapper)**

直接与 GDAL API 交互，所有方法设计为线程安全。

* **数据集 I/O 引擎**：安全管理栅格数据的打开、创建、内存分配与指针释放。  
* **栅格转换引擎**：封装 GDALWarp 和 GDALCreateCopy，处理重采样、投影转换及格式压缩。  
* **金字塔构建器**：封装 GDALBuildOverviews，专为 COG 等需要预览优化的格式服务。  
* **回调拦截器**：实现 GDALProgressFunc 获取进度，重载 CPLErrorHandler 结构化底层报错。

### **3.4 配置与持久层 (Configuration & Persistence Module)**

管理软件状态，提供预设驱动逻辑。

* **预设管理器**：负责用户转换模板的 JSON 读写与出厂预设加载。  
* **全局偏好设置**：管理并发线程限制、默认路径等应用级参数。

### **4.2 逻辑流转机制**

1. **初始化**：软件启动，优先读取本地 JSON 预设文件；若无则加载 Qt 资源文件中的硬编码出厂预设。  
2. **状态绑定**：UI 下拉框选中预设，面板参数自动回显；用户修改参数即标记为临时自定义状态。  
3. **任务生成**：点击“加入队列”，UI 将当前配置**深拷贝**并生成独立 Task 实例压入调度层，确保后续预设的切换不会污染已在队列中的任务。

**5\. 推荐的项目文件结构**

为了支撑上述架构，推荐采用标准的现代 C++ CMake 目录布局。这种结构严格对应了四大核心模块，清晰且易于扩展。
